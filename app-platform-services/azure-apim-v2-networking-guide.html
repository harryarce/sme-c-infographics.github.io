Public & Private Networking — explained simply

A friendly walkthrough of how Standard v2 and Premium v2 deploy, where they live on the network, and exactly how to set each one up in the Azure portal. No deep networking background required.

Words We Will Use

Four ideas to know first

Before we look at deployment scenarios, let's level-set on the building blocks. Each one shows up later in this guide.

🌐

Public

The APIM endpoint is internet-accessible. Even when APIM has VNet integration for outbound, the gateway and developer portal endpoints stay publicly reachable. You secure it using the APIM ip-filter policy.

🚪

Internal (Private)

The APIM endpoint is closed to the public internet — only reachable through your private network. A Private Endpoint uses a VNet IP, traffic flows over Private Link on the Microsoft backbone, eliminating public exposure.

🔌

Integration

VNet integration means APIM connects outbound to private backends through a VNet, but the gateway itself stays public (unless you add a Private Endpoint). Both Standard v2 and Premium v2 support this.

Outbound only
🛡️

Injection

Full VNet injection means APIM lives inside a VNet — the gateway is reachable at a private IP only. Best for full network isolation. Only Premium v2 supports injection. Standard v2 cannot be injected.

Premium v2 only
Network Architecture Progression

From fully public to fully private

Think of these as four points on a spectrum. Each step adds isolation — and complexity.

1

Public Gateway

Default internet endpoint with optional IP filtering.

2

Outbound VNet Integration

APIM reaches private backends while remaining publicly reachable.

3

Inbound Private Endpoint

Gateway becomes reachable only from your private network.

4

Full VNet Injection

APIM lives entirely inside the VNet with a private IP only.

The Four Deployment Scenarios

Pick the model that matches your need

Each scenario lists when to use it, what's good and bad about it, plus a step-by-step setup summary based on the Azure portal flow.

Scenario 1

Standard v2 — Public

Internet-facing gateway, locked down with an IP restriction policy. The simplest place to start.

🌍 Internet → 🔒 ip-filter Policy → ⚡ APIM (public URL) → 🎯 Backend

★ When to use

  • Publishing APIs to external developers or partners
  • Cost-efficiency matters, full isolation is not required
  • Backends are reachable on the public internet or through service-level firewalls
  • You want the fastest path from zero to a working gateway

✓ Pros

  • Simplest, fastest to deploy
  • Lowest-cost SKU in the v2 family
  • ip-filter policy works at global, workspace, product, API, or operation scope
  • No VNet, subnet, or NSG planning required

✕ Cons

  • Gateway is reachable from the internet by default
  • No NSG rules — only IP filter policy
  • Not suitable for strict private-only requirements

Step-by-step setup (Azure portal)

Result: a Standard v2 instance with a public gateway URL, locked to a specific IP via the ip-filter policy.

1
Create the resource

Azure portal → Create a resource → search API Management → Create.

2
Basics tab — service details

Subscription, Resource group (e.g., demo-apim-rg), Name (e.g., demo-apim-public), Region, and Pricing tier = Standard v2.

3
Networking tab — keep defaults

Leave public network access enabled. APIM will get a public URL like https://demo-apim-public.azure-api.net.

4
Review + Create

Click Review + create, then Create. Wait for the deployment-complete notification.

5
Add a sample API

Open the instance → APIs → + Add API → choose the built-in Echo API → leave defaults → Create.

6
Apply IP restriction policy

APIs → All APIs → click the inbound processing </> code editor. Inside <inbound>:

<ip-filter action="allow">
  <address>1.2.3.4</address>
</ip-filter>

Replace 1.2.3.4 with your client IP. With action="allow", any IP not listed is automatically denied. Use <address-range from="x.x.x.x" to="y.y.y.y" /> for ranges. Click Save.

How to test

From the allowed IP, hit:

https://demo-apim-public.azure-api.net/echo/resource

You should get a successful response. From any other IP, APIM returns HTTP 403 Forbidden.

In one line: A public gateway you can spin up in minutes, then lock down with IP allow-lists. Best fit for partner or developer APIs that don't require network-level isolation.
Scenario 2

Standard v2 — Internal (Private Endpoint + VNet Integration)

Inbound Private Endpoint + outbound VNet integration. End-to-end isolation without paying for Premium injection.

🏢 Internal Apps → 🚪 Private Endpoint → ⚡ APIM → 🔌 VNet Integration → 🎯 Private Backend

★ When to use

  • APIs are for internal apps only — no public exposure
  • Backends live in a VNet, on-prem (via ExpressRoute), or another private network
  • You need private-IP inbound but want Standard v2 pricing
  • You can accept that the underlying platform is multi-tenant (vs. injection)

✓ Pros

  • No public inbound surface once Private Endpoint is wired up
  • Reach private backends through outbound VNet integration
  • Cheaper than Premium v2 isolation options
  • Public access can be disabled after Private Endpoint is created
  • End-to-end isolation between clients and backends

✕ Cons

  • Inbound NSG rules do not apply on integration (only injection)
  • No full VNet injection — the gateway itself isn't in your VNet
  • DNS resolution must be planned (private DNS zones)
  • Two subnets needed: one for integration, one for the Private Endpoint
Why not injection? Standard v2 doesn't support full VNet injection — only Premium v2 does. Standard v2 uses Integration (outbound) + Private Endpoint (inbound) to achieve a similar result.

Step-by-step setup (Azure portal)

Result: a Standard v2 instance reachable only via Private Endpoint, with outbound traffic flowing through your VNet.

1
Create a Virtual Network with two subnets

Create demo-vnet with address space 10.0.0.0/16 (same region as APIM).

Subnet A — apim-integration-subnet: 10.0.0.0/24 (min /27, /24 recommended). Delegate to Microsoft.Web/serverFarms.
Subnet B — apim-private-endpoint-subnet: 10.0.1.0/27. No delegation. The Private Endpoint subnet cannot be delegated.

2
Create an NSG for the integration subnet

Name it apim-integration-nsg in the same region.

3
Attach the NSG and add outbound rules

Open the NSG → Subnets → Associate with apim-integration-subnet. An NSG must be associated with the integration subnet.

Add two outbound rules — both TCP/443, Action: Allow:
• AllowStorage → Service Tag Storage, priority 100
• AllowKeyVault → Service Tag AzureKeyVault, priority 110

Inbound NSG rules are not enforced for v2 integration. To enforce inbound NSG, use injection (Scenario 4).

4
Create the APIM instance with Private Endpoint + Integration

Create resource → API Management → Pricing tier Standard v2.

Networking tab → choose Inbound private link and/or outbound virtual network integration.
• Private Endpoint: Name apim-private-ep, Sub-resource Gateway, VNet demo-vnet, Subnet apim-private-endpoint-subnet, integrate with private DNS zone privatelink.azure-api.net (default).
• Outbound VNet Integration: select demo-vnet + apim-integration-subnet.

5
Disable public network access

You can only do this after the Private Endpoint is configured. Instance → Deployment + infrastructure → Network → Edit → set Public network access = Disabled → Save.

6
Verify DNS

Azure auto-creates a record in privatelink.azure-api.net pointing the APIM hostname to its private IP. Confirm under the Private Endpoint → DNS configuration. Make sure your VNet is linked to the private DNS zone.

7
Add a sample API

APIs → + Add API → import the Echo API. To verify outbound, set the Web service URL to a backend inside demo-vnet or a peered network.

How to test

From a VM inside demo-vnet:

https://demo-apim-internal.azure-api.net/echo/resource

Should succeed over the private network. From a machine outside the VNet, the URL should not load — proving the gateway is internal-only. Reminder: each APIM instance supports up to 100 Private Link connections, and only the Gateway sub-resource supports inbound Private Link.

In one line: The semi-private sweet spot — private inbound and private outbound, without paying for full injection. Great for internal-only APIs on a budget.
Scenario 3

Premium v2 — Public

Internet-facing Premium gateway with higher scale (up to 30 units), availability zones, and the ability to add Private Endpoint later.

🌍 Internet → 🔒 ip-filter Policy → ⚡ APIM Premium → 🎯 Backend

★ When to use

  • You need higher scale-out (up to 30 units) than Standard v2 offers
  • You want availability-zone support or premium features
  • You may evolve into private inbound later via Private Endpoint
  • SLA, multi-region, or advanced policy needs justify Premium pricing

✓ Pros

  • Higher scale and richer feature set than Standard v2
  • Public by default, but can add Private Endpoint inbound
  • Outbound VNet integration available for private backends
  • ip-filter policy works identically to Standard v2

✕ Cons

  • Higher cost than Standard v2
  • VNet injection cannot be turned on later — only at creation (see Scenario 4)
  • Still public-facing unless you actively add Private Endpoint
  • You can't switch between injection and integration once chosen
When to choose Premium v2 over Standard v2 for public: when you need full VNet injection later, higher scale (up to 30 units), or availability-zone support. Otherwise Standard v2 gives the same public functionality at a lower cost.

Step-by-step setup (Azure portal)

Result: a Premium v2 instance with a public gateway URL, locked to a specific IP via the ip-filter policy.

1
Create API Management — Premium v2

Create resource → API Management → fill Subscription, Resource group, Name, Region. Set Pricing tier = Premium v2.

2
Networking tab — public access

Keep default Public connectivity (no VNet injection, no Private Endpoint). Click Review + create → Create. Premium v2 may take some time to deploy.

3
Add a sample API

After deployment, go to APIs and add the Echo API (same as Scenario 1, Step 5).

4
Apply IP restriction policy

Repeat Scenario 1 Step 6 — add the <ip-filter action="allow"> snippet inside <inbound> and Save. The policy works identically across all v2 tiers.

How to test
https://<your-premium-apim-name>.azure-api.net/echo/resource

From the allowed IP it should succeed. Any other IP gets HTTP 403 Forbidden.

In one line: Premium scale and features without full lockdown — flexible for high-throughput public APIs that may grow into a private posture over time (via Private Endpoint, not injection).
Scenario 4

Premium v2 — Internal (Full VNet Injection)

The most secure configuration. APIM lives entirely inside your VNet with a private IP only — no public endpoint at all.

🏢 Private Network → 🛡️ NSG (inbound rules apply) → ⚡ APIM (inside VNet) → 🎯 Private Backend

★ When to use

  • Compliance, regulatory, or security policy mandates no public IP at all
  • You need NSG enforcement on inbound traffic, not just ip-filter
  • APIM and backends must both stay inside the corporate network boundary
  • You can lock the deployment model in at creation time (it's irreversible)

✓ Pros

  • Fully private — no public IP exposure
  • NSG rules apply for full network-level inbound control
  • Highest level of isolation v2 offers
  • Premium scale (up to 30 units) and feature set

✕ Cons

  • Injection can only be set at creation — not switchable later
  • Cannot switch between injection ↔ integration on the same instance
  • Highest-cost option in the v2 family
  • Requires careful subnet planning (≥ /27 minimum, /24 recommended)
  • You must manage your own DNS — public DNS won't resolve the gateway
  • External developers can't reach the gateway without extra plumbing
📌 Critical timing rules: You can only inject Premium v2 into a VNet at creation time. You cannot inject an existing instance later. You also cannot switch between injection and integration once an instance is created.

Step-by-step setup (Azure portal)

Result: a Premium v2 instance with a private IP only, inside your VNet, with NSG-enforced inbound rules and a self-managed DNS record.

1
Create a Virtual Network and injection subnet

Create demo-vnet2 in the same region/subscription as your APIM, address space 10.1.0.0/16.

Subnet — apim-injection-subnet: 10.1.0.0/24 (min /27, /24 recommended). Delegate to Microsoft.Web/hostingEnvironments (different from the integration scenario).

The injection subnet can only be used by one APIM instance and cannot be shared.

2
Create an NSG for the injection subnet

Name it apim-injection-nsg in the same region.

3
Attach NSG and add outbound rules

NSG → Subnets → Associate with apim-injection-subnet. NSG association is mandatory.

Add two outbound rules — TCP/443, Action: Allow:
• AllowStorage → Service Tag Storage, priority 100
• AllowKeyVault → Service Tag AzureKeyVault, priority 110

You can additionally restrict outbound to internet and only allow VNet/peered destinations.

4
Create APIM (Premium v2) with VNet Injection

Create resource → API Management → Pricing tier = Premium v2.

Networking tab:
• Connectivity type = Virtual network
• Type = Virtual Network injection
• Select demo-vnet2 + apim-injection-subnet

Click Review + create → Create. Deployment may take 30+ minutes.

5
Note the private IP

From the APIM Overview page, capture the assigned private IP (e.g., 10.1.0.4).

The IP is dynamically assigned — you can't predict it before deployment, and switching subnets can change it.

6
Set up DNS for the private hostname

Public DNS won't resolve the gateway. Recommended:

• Create an Azure Private DNS Zone for azure-api.net
• Link the zone to demo-vnet2
• Add an A record mapping demo-apim-injected → the APIM private IP

Quick demo alt: edit the hosts file on a VM inside the VNet:
10.1.0.4  demo-apim-injected.azure-api.net

If you use a custom DNS resolver, also ensure name resolution for *.vault.azure.net (Key Vault). Azure Private DNS handles this automatically.

7
Add a sample API

APIs → + Add API → import the Echo API. Set the backend URL to a service accessible inside demo-vnet2 (or via peering).

How to test

From a VM inside demo-vnet2:

nslookup demo-apim-injected.azure-api.net
curl https://demo-apim-injected.azure-api.net/echo/resource

nslookup should return the private IP. curl should succeed. From outside the VNet, DNS won't resolve and the call should time out — proving full isolation.

In one line: Maximum isolation. Full VNet injection is a one-time, irreversible decision — make it deliberately when full network isolation is mandatory.

Security & Access Controls

Controls you can layer on

Different scenarios unlock different controls. Here's what's available — and where it applies.

🔐 IP Filter Policy

Enforces allow/deny rules for public endpoints. Works on all v2 tiers and can be scoped at global, workspace, product, API, or operation level.

🚪 Private Endpoint

Removes inbound internet exposure entirely. Available on both Standard v2 and Premium v2. Only the Gateway sub-resource is supported.

🛡️ Inbound NSG Rules

Apply only when full VNet injection is in place — meaning Premium v2 injected at creation time. Inbound NSG does not apply to integration on either tier.

🚫 Disable Public Access

Once a Private Endpoint is created, you can turn off public network access from Deployment + infrastructure → Network for clean private-only posture.
