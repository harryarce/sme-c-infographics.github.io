How this site schedules accuracy work

A scheduled loop creates review issues, gives Copilot enough context to propose fixes when assignment is configured, and keeps humans in the approval seat. The automation reduces discovery work; it does not guarantee a correct PR.

The loop

Five steps from cron tick to review issue

The durable handoff is the GitHub issue. If Copilot assignment is available and a user-scoped PAT is configured, that issue can start agent work; otherwise it remains visible for a human to assign or handle.

1

Scheduled trigger

accuracy-review.yml runs weekly on Thursdays. copilot-page-review.yml runs monthly on the 1st.

2

Review bodies built

Scripts emit issue bodies from stale accuracy metadata or audit findings, with per-run caps and duplicate checks.

3

Issues opened

Weekly issues use accuracy-review; monthly issues use copilot-review. Both include hidden page markers for deduplication.

4

Assignment attempted

If COPILOT_ASSIGN_TOKEN exists and Copilot is assignable, the workflow assigns Copilot. Without the PAT, issues are intentionally left unassigned.

5

PR remains reviewed

Copilot may open a draft PR with corrections and citations. Humans still merge, close, or ask for another round.

Why this shape works

  • Zero manual scheduling. Cron finds stale pages and audit findings without someone remembering to start the review cycle.
  • Microsoft Learn is the expected source family. The weekly issue prompt asks Copilot to verify against current Learn documentation; Learn MCP can help when configured, but the workflow does not depend on MCP.
  • Failure is visible. Without COPILOT_ASSIGN_TOKEN, the workflow creates unassigned issues instead of pretending Copilot started.
  • Humans still approve. Nothing auto-merges. Every PR needs a reviewer to accept or reject the fact-check.

What it doesn't do

  • Not guaranteed PR creation. The workflows create issues. Copilot PRs require assignment to trigger, agent availability, and a task that results in edits.
  • Not perfect. Copilot can miss subtle inaccuracies or flag false positives. Review quality still depends on the reviewer.
  • Not fully portable as-is. This pattern fits public documentation-heavy content. Other domains may need different source rules and reviewers.
  • Not a substitute for SME review. A correctly cited stale fact is still stale if the source lags the live product.

About this infographic

This is the lived example, not a pitch. The workflow described here runs on this repository. The claim is narrow: weekly freshness checks and monthly page audits turn accuracy work into visible GitHub issues, and can trigger Copilot drafting when PAT-based assignment and agent access are configured.
