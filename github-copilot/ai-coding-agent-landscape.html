Which AI coding agent fits which team?

Not a ranking. A decision tree for choosing the control plane first: GitHub-native (web or new desktop app), GitHub-hosted third-party, standalone vendor workflow, or Cursor’s cloud surface.

Accurate as of June 27, 2026 · Async GitHub-connected agents, not local IDE copilots

Three questions in order

Most teams short-circuit at question 1 or 2. If you're still undecided after question 3, your real constraint is usually control plane, model preference, or existing developer workflow.

Q1 · Is GitHub.com your primary code host?
No
Stop here for this series. The GitHub-hosted agents and GitHub Actions patterns assume GitHub repos. Your shortlist is local IDE/CLI agents or a vendor cloud agent that supports your code host.
Yes
Continue to Q2.
Q2 · Do you want the agent managed inside GitHub/Copilot?
Yes
Start with Copilot cloud agent, then evaluate GitHub-hosted Claude or Codex when model choice matters. These agents share GitHub entry points, policies, and review loops.
No
Continue to Q3. You are choosing a standalone vendor workflow, where dashboard, billing, setup, and audit surfaces may live outside GitHub.
Q3 · What matters most?
Specific model
Use GitHub-hosted Claude/Codex if you want model choice inside GitHub; use standalone Claude/Codex if you need that vendor's own workflow or account controls.
Lowest repo change
Copilot or GitHub-hosted Claude/Codex avoid workflow YAML. Standalone Codex/Cursor also avoid workflow files, but move more setup into vendor settings.
Custom CI workflow
Claude Code Action is the clean fit when you want a GitHub Actions workflow, your own runner posture, and explicit secret/permission control.
Editor ↔ cloud parity
Cursor Cloud Agents fit teams already using Cursor and wanting the same agent surface from editor, web, Slack, Linear, or GitHub.
Honest fit

When each control plane fits

Every option below can be the right answer. The useful question is where you want policy, billing, setup, and review state to live.

GitHub Copilot cloud agent

GitHub-native · Pro, Pro+, Max, Business, Enterprise
Best fit: GitHub is the control plane, Copilot is already approved, and platform wants low-friction issue-to-PR adoption across many repos.
Watch-outs: Copilot cloud agent still uses Actions minutes and AI credits, and it does not honor Copilot content exclusions.
  • Zero per-repo files required
  • First-party GitHub agent identity
  • GitHub agent entry points and review loop
  • Uses Actions minutes + AI credits

GitHub-hosted Claude or Codex

Third-party agents · GitHub/Copilot surface
Best fit: you want Claude or Codex model choice while keeping GitHub issues, PR comments, policies, and agent management as the primary workflow.
Watch-outs: third-party agents are public preview and require Copilot plan eligibility plus policy enablement.
  • Supported agents: Anthropic Claude and OpenAI Codex
  • Agents tab, issue assignment, PR comments, Mobile, VS Code
  • Same protections and limitations as Copilot cloud agent
  • Uses Actions minutes + AI credits

Standalone Claude / ChatGPT Codex

Vendor workflow · App, settings, or Actions
Best fit: you already standardize on Claude Code or ChatGPT Codex and want that product's own settings, review behavior, or automation model.
Watch-outs: setup and audit state can split between GitHub and vendor settings, especially for standalone Codex review or Claude workflow secrets.
  • Claude Code Action uses workflow YAML + secret
  • ChatGPT Codex review uses Codex cloud settings
  • @claude and @codex review patterns are both documented
  • Good for vendor-specific automation beyond GitHub-hosted agents

Cursor Cloud Agents

External SaaS · Cursor team plan
Best fit: Cursor IDE shop wanting one agent experience across editor, web, Slack, Linear, and GitHub.
Watch-outs: Cursor’s value is strongest when the team is already bought into Cursor; regulated teams should review repo access, billing, privacy, and audit requirements.
  • Triggered from Cursor web, Desktop, Slack, Linear, GitHub, API
  • Cloud agents run away from the local laptop
  • MCP + hooks support (.cursor/hooks.json)
  • Team/Enterprise plans add admin and governance controls

GitHub Copilot desktop app NEW

GitHub-native · macOS/Windows desktop · Public preview
Best fit: developers who want an agent-native desktop surface for parallel work — Agent Merge for CI/review babysitting, worktrees for multi-task isolation, Rubber Duck review for second-opinion critique, Canvases for visual planning — with multi-model choice (OpenAI, Anthropic, Google) on a GitHub identity.
Watch-outs: public preview; behavior, policy surface, and billing details are still evolving. Pair with existing repo policies (rulesets, content exclusion) until enterprise controls catch up.
  • Agent Merge, worktrees, Rubber Duck review, Canvases
  • Multi-model: OpenAI, Anthropic, Google
  • Source: github/app
  • Reuses Copilot subscription + GitHub identity

What this piece is not

  • Not a capability ranking. None of these cards claim one agent writes better code than another; that depends on task, repo, model, and benchmark.
  • Not a procurement recommendation. Customers still need their own legal, privacy, billing, and InfoSec review.
  • Not one integration path per vendor. Claude and Codex now have both GitHub-hosted and standalone patterns.
