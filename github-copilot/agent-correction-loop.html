When the agent's PR is wrong

Every async coding agent ships broken first drafts sometimes. The question is how you tell it to try again, which control plane owns the next round, and whether the review trail stays easy to reconstruct.

Shape of the loop

Four phases, same order

Every async agent cycles through the same four phases. What differs is where state lives at each phase - in GitHub issues and PRs, in Actions logs, or in a vendor workspace.

The loop

Four tools, four correction shapes

What a reviewer has to do to get a v2. The cheaper and more inspectable this loop is, the more viable the "assign it to the agent" pattern becomes for real work.

GitHub Copilot cloud agent

Native PR surface
1
Comment in the PR. @copilot address the null-check on line 42.
2
Agent wakes on the same branch. New commit pushed, same PR.
3
Review comments resolve inline. Standard GitHub review UI.
!
Watch the limits. Runs can consume Actions minutes and Copilot premium requests, and content exclusions are not honored by the cloud agent.

GitHub-hosted Claude / Codex

Third-party agents in GitHub
1
Use the GitHub agent surface. Ask from supported GitHub issues or PR workflows instead of switching to a vendor dashboard first.
2
GitHub-hosted run starts. The agent works in a GitHub-managed environment and posts results back through GitHub.
3
Review stays where the code review lives. Follow-up feedback can stay attached to the issue or PR.
!
Same cloud-agent limits apply. GitHub says third-party agents share Copilot cloud agent protections and limitations.

Claude Code Action

@claude-triggered workflow
1
Comment with @claude. Workflow fires on the event.
2
Action runs in GitHub Actions. Progress is visible through the workflow run and PR comments.
3
New commit on the PR branch. Same PR, one Actions run per trigger.
!
Each round has runner and model cost. You manage workflow YAML, app permissions, and Anthropic credentials.

Standalone Codex / Cursor

Vendor cloud workspace
1
Start from GitHub or the vendor product. Codex and Cursor both support GitHub-connected review/task flows.
2
Agent works in a vendor-hosted environment. The product owns much of the run orchestration and task state.
3
Outputs come back to GitHub. Commits or review results can still land on the PR.
!
Reviewers may need a second surface. Detailed task history or continuation can live in ChatGPT Codex, Cursor web, Cursor Desktop, Slack, Linear, or API-driven flows.
Practical pattern

What a good correction comment looks like

All four respond better to concrete, diff-specific feedback than to "this is wrong, try again." Pattern below works on any of them.

Weak
@copilot this doesn't work, please fix it
Better
@copilot the rate-limit check in auth.ts:42 uses user.id as the key but that's null for unauthenticated requests. Use request.ip as the fallback and add a test for the anonymous case.

What makes a good correction loop

  • One routine surface. Reviewers should be able to give normal correction feedback without leaving the issue or PR.
  • Visible iteration cost. Teams should know whether a "try again" consumes Copilot premium requests, Actions minutes, vendor tokens, or subscription quota.
  • State in the review trail. Future maintainers reading git log and PR comments should understand why the final patch looks the way it does.
  • Stop after 2-3 rounds. If it's still wrong, take over manually; iteration cost exceeds writing the fix.

What this piece doesn't compare

  • Not iteration quality. No claim about which tool's v2 is better than its v1.
  • Not response latency. All four run async; seconds-to-response varies by load, not product.
  • Not custom prompts. Each tool has fine-grained prompt-tuning features; this piece only covers the default review-driven loop.
  • Not trigger-phrase overrides. Agent names and triggers can vary by configuration; this piece compares the default public workflow shapes.
