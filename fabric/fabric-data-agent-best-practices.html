Semantic Model

Prep for AI — Three Pillars

Power BI's "Prep for AI" feature has three components that directly impact how the data agent interprets your semantic model. Click each tab to explore.

🎯 What It Does

Defines a focused subset of your model — tables, columns, and measures — that the AI should prioritize when generating DAX queries.

  • Reduces ambiguity & improves accuracy
  • Decreases response latency
  • The DAX tool uses this schema, not the full model

💡 Tips

  • Select the same tables in both Prep for AI and the data agent
  • Include dependent objects — if a measure references other measures/columns, include them all
  • Use Semantic Link LabsThe get_measure_dependencies function helps you identify all dependencies for a measure. to find dependencies
  • Use the Power BI Modeling MCP server to auto-generate business-friendly names

📋 Example

Problem: Model has Total Revenue, Gross Sales, Net Sales, Sales After Returns. User asks "What were our sales last quarter?" — AI guesses wrong.

Fix: Configure schema to include only Net Sales (the metric the team actually uses). The AI no longer has to guess.

✅ What It Does

User-approved visual responses that trigger for specific questions. Stored at the semantic model level, they work across any agent using the same model.

  • Checks for exact or semantic match before generating a new response
  • Uses visual properties (columns, measures, filters) to guide DAX generation

💡 Configuration Tips

  • Use 5–7 trigger questions per verified answer to cover natural phrasing variations
  • Include both formal and conversational phrasings
  • Configure up to 3 filters for flexible slicing
  • If you rename objects, update & re-save verified answers
  • Don't reference hidden columns — verified answers won't work with them

📋 Example

Problem: User asks "Show me performance by territory." AI interprets "territory" as a product category column.

Fix: Create a verified answer using a regional sales visual with triggers like "sales by territory", "performance by region", etc.

📝 What It Does

Unstructured guidance on the semantic model that clarifies terminology, analysis approaches, and business context the AI wouldn't otherwise know.

  • Interpreted by the LLM — clear & specific is better than complex
  • Only Prep for AI instructions reach the DAX tool — agent-level instructions do not

💡 Effective Patterns

  • Time periods: "Peak season = November–January"
  • Metric preferences: "For profitability, use Contribution_Margin, not Gross_Profit"
  • Routing: "For inventory, use Warehouse_Inventory over Sales_Orders"
  • Defaults: "Analyze revenue by fiscal quarter unless otherwise specified"

📋 Example

Problem: User asks "Who were the top performers last month?" AI doesn't know what "top performer" means internally.

Fix: Add instruction: "A top performer is a sales rep who achieves ≥ 110% of monthly quota. Use Rep_Performance, filter Quota_Attainment ≥ 1.1"

Configuration

Data Agent Configuration Best Practices

Agent-level and data-source-level instructions control how the agent interprets queries, selects sources, and formats responses.

1   Create Specialized Agents for Specific Domains▼
Design agents focused on a specific domain or use case rather than trying to handle everything. A narrow focus lets you optimize instructions, select relevant sources, and use domain-specific terminology — making responses more accurate and reliable.

Example: Instead of a general "customer Q&A" bot, build an agent tailored for the leadership team that combines CRM, finance, and project data for meeting prep.
2   Minimize Data Source Scope▼
Include only the data sources, tables, and columns necessary to answer expected questions. A focused scope = more accurate and efficient queries.

Tip: Limit to 25 or fewer tables per data source for optimal results.
3   Be Specific — Say What To Do, Not Just What Not To Do▼
Provide clear guidance on the correct approach rather than only listing prohibitions.

Less effective: "Do not provide outdated pay information."
Better: "Always use the most recent pay records from the official payroll system. If missing, inform the user and suggest they contact HR."
4   Define Business Terms, Abbreviations & Synonyms▼
Clarify anything ambiguous or organization-specific:
  • Similar concepts: "calendar year" vs. "fiscal year"
  • Business terms: "quarter", "SKU", "NPS"
  • Abbreviations: "MAU" = Monthly Active Users

Where: Cross-source definitions → agent-level instructions. Source-specific definitions → data-source instructions.
5   Use Leading Words to Nudge Query Generation▼
Include SQL/DAX/KQL syntax fragments in data-source instructions to guide the model:

Less effective: "Find products containing 'bike'."
Better: "Find products containing 'bike' — LIKE '%bike%'"

These hints help the model recognize the correct pattern-matching, filter, or join logic.
6   Write Clear, Focused Instructions — Skip the Noise▼
Be concise and purposeful. Avoid vague, outdated, or overly broad content.
  • Define clear scope & responsibilities
  • Reference the correct data source without technical overload
  • Specify fallback behavior for missing data
  • Set tone & style
7   Write Detailed Agent-Level Instructions▼
Describe the agent's role, expected behavior, tone, and how it handles different query types. Include use cases, preferred data sources, and fallback behavior.

Litmus test: "Would someone unfamiliar with these data sources know which to use and how?" If not, add the missing context.

Structure with sections: Tone & Style, General Knowledge, Data Source Descriptions, When Asked About… routing.
8   Provide Detailed Data Source Instructions▼
Guide the agent in forming accurate queries by defining:
  • Purpose of the data source and which questions it answers
  • Required columns to include in responses
  • Join logic between tables
  • Typical value formats (abbreviations vs. full names)
  • Example values: e.g., State uses "CA", "NY", "TX"

Remember: The agent can't see individual row values before executing — include value examples to guide filtering.
9   Use Example Queries for Complex Logic▼
Example queries act as templates the agent generalizes from. For each user question, the agent performs a vector similarity search to retrieve the top 3 most relevant examples.

Focus on queries involving:
  • Filtering & joins
  • Aggregations
  • Date handling

Tip: A well-formed query is often clearer than paragraphs of text trying to explain the same logic.

Implementation

Recommended Workflow

Follow these steps in order for the best results. Each step builds on the previous one.

1

Optimize the Semantic Model

Use Best Practice Analyzer and Semantic Model Memory Analyzer to fix data types, remove unnecessary columns, and improve DAX efficiency. Add descriptions to all objects in your AI schema.

2

Define AI Data Schema

In Prep for AI, select only the tables, columns, and measures relevant to your agent's scope. Include all dependent objects.

3

Create Verified Answers

Identify your most common questions and configure verified answers with 5–7 trigger question variations each.

4

Add Semantic Model to Agent & Test

Before adding AI instructions, test agent responses to understand where instructions are actually needed.

5

Add AI Instructions in Prep for AI

Based on testing, define business terminology, analysis preferences, and data source priorities. Keep them in Prep for AI — not at the agent level.

6

Prepare Report Visuals

Ensure visuals across connected reports have descriptive titles. The AI uses visual metadata (title, columns, measures, filters) to ground responses.

7

Verify & Review DAX

When testing, review the generated DAX in each response. If results are wrong, analyze the DAX to identify which configuration needs adjustment.

8

Configure Agent-Level Instructions

Add only cross-source guidance: response formatting, routing rules, abbreviations, tone. Do not add semantic-model-specific instructions here.

9

Validate, Iterate & Deploy

Use the Fabric Data Agent Python SDK for automated evaluations. Involve stakeholders. Use Git integration & deployment pipelines for ALM across dev → test → prod.

Avoid

Common Pitfalls

These issues are the most frequent causes of inaccurate or slow data agent responses.

Not Using Star Schema

Flat or denormalized tables make DAX less efficient. DAX is optimized for star schema with clear fact and dimension tables. Unpivot wide tables into normalized structures.

Relying on Hidden Fields

Verified answers won't work if they reference hidden columns. Ensure all referenced objects are visible.

Including Unnecessary Measures

Helper measures and intermediate objects add noise. Include only measures that calculate actual business metrics.

Duplicate / Overlapping Measures

Total Sales, Sales Amount, Revenue all calculating similar things = ambiguity. Consolidate or clearly differentiate and exclude duplicates.

Non-Descriptive Naming

Names like TR_AMT, F_SLS, DIM_GEO_01 give the AI zero context. Use business-friendly names, or at minimum provide descriptions and synonyms.

Relying on Implicit Measures

Implicit measures produce unpredictable results. Create explicit DAX measures and set correct default summarization on numeric columns.

Ambiguous Date Fields

Multiple date columns without guidance confuse the AI. Use verified answers and AI instructions to specify which date field to use by default.

Conflicting Instructions

AI instructions contradicting verified answers create unpredictable behavior. Keep them aligned.

Skipping Schema Refinement

Large models with many similarly named fields need focused AI data schemas. Don't skip this step.
