1 The challenge

Per-tenant analytics doesn't scale by hand

An ISV embedding analytics must give every customer their own semantic model, reports, refresh schedule, security boundary, and branding. Multiply that by thousands of tenants and the manual approach collapses under its own weight.

◆ One ISV analytics product
Tenant A — Contosomodel · report · refresh
Tenant B — Fabrikammodel · report · refresh
Tenant C — Northwindmodel · report · refresh
Tenant D — Tailwindmodel · report · refresh
+ 9,996 more tenants · each a full lifecycle to manage
  • ⧉
    Duplication explosionCloning and hand-editing models & reports for every customer is slow, error-prone, and impossible to keep consistent.
  • ↻
    Drift & inconsistencyA measure fix or schema change must reach every tenant — manually, that guarantees version drift.
  • ⚙
    Operational loadDeploys, refreshes, permissions, and governance multiply linearly with tenant count.
  • ⚇
    Isolation & capacityEach tenant needs its own security boundary while sharing capacity efficiently — hard to do safely by hand.
2 The platform stack

Six first-party capabilities, each precisely scoped

Generic talk of "Power BI & Fabric MCP" hides important distinctions. Each component below is a distinct, documented Microsoft capability with its own scope, transport, and auth model — and they compose.

⬡
The semantic-model engine

Power BI Modeling MCP ServerPREVIEWLOCAL · STDIO

A local MCP server (@microsoft/powerbi-modeling-mcp) that connects AI agents to Power BI semantic models in Desktop, a Fabric workspace, or PBIP files. Executes modeling operations only — tables, columns, measures, DAX, relationships, RLS, calculation groups, translations — with bulk, transactional edits across hundreds of objects. It cannot author report pages or visuals.

Auth: interactive · service principal Modes: readwrite · readonly

The platform-control plane

Fabric Core MCP ServerPREVIEWREMOTE

A remote endpoint at api.fabric.microsoft.com/v1/mcp/core that turns prompts into Fabric REST API calls — respecting your RBAC and writing audit logs. Tools cover catalog search, workspace CRUD, item management (lakehouses, semantic models, notebooks, pipelines), permission & folder management, and capacity info.

Auth: OAuth 2.0 · Entra ID Respects: Fabric RBAC + audit

Skills for Fabric — PowerBI authoring plugin

Semantic Model & Report Authoring skillsPREVIEW

The Power BI Report Authoring skill (powerbi-report-authoring) authors the report layer — pages, visuals, slicers, themes — in PBIR/PBIP, paired with the Report Design & Report Planner skills.

The foundational surface

Fabric REST APIs

The programmable foundation every layer above ultimately calls. Direct REST gives you maximum control: item-definition CRUD, deployment pipelines, Git integration, capacities, and long-running operations — drivable by the Fabric CLI, the fabric-cicd Python library, or the Terraform provider for Fabric in your own CI/CD.

CI/CD: fabric-cicd · Terraform · CLI

Maximum control + reusability

Your own custom MCP Server

Because MCP is an open specification, an ISV can build a bespoke MCP server that wraps the Fabric REST APIs plus your own provisioning logic — exposing tenant-shaped tools like provision_tenant or rotate_tenant_keys. You get REST-level control packaged as reusable, governable tools any agent can call.

3 Architecture & ownership

Skills teach. MCP servers do. REST is the floor.

Straight from Microsoft's documentation: skills supply the expertise; MCP servers supply the live data connection; the REST APIs are what both ultimately drive.

◈

Skills for Fabric

Knowledge & patterns — markdown loaded into the agent's context

  • Teach the agent what to do & which APIs to call
  • Static instructions (SKILL.md) — they don't execute
  • Encode best practices & operational patterns
  • Open source, MIT — extend or fork them
+
⬡

MCP servers

Live data access — executable servers run as separate processes

  • Actually do the operation against your model/tenant
  • Strongly typed, deterministic, mapped 1:1 to Fabric concepts
  • Respect your existing RBAC & permissions
  • Modeling MCP (local) & Core MCP (remote) are complementary
“Skills provide the expertise and MCP servers provide the data connection. They both work best together.”— Skills for Fabric overview, Microsoft Learn

4 The spectrum of control

Abstraction where you want speed, control where you need it

Higher layers ship faster with opinionated best practices; lower layers deliver total control and customization. ISVs running Fabric at scale operate all four — and wrap the bottom in a custom MCP server to capture both.

◀ More guidance & speed · higher abstractionMore control & customization · lower level ▶
LAYER 1

Skills for Fabric

Agent-guided · opinionated

Natural language in, governed Fabric objects out — with best practices baked in. Fastest path to a working result.

Best forCommon authoring & lifecycle tasks where Microsoft's patterns fit.
LAYER 2

Prebuilt MCP servers

Deterministic · governed

Modeling MCP & Core MCP expose typed, safe tools that map 1:1 to Fabric concepts and honor RBAC.

Best forPrecise, auditable model & platform operations at scale.
LAYER 3

Fabric REST APIs

Full surface · you orchestrate

Call any endpoint directly — item definitions, deployment pipelines, Git — from fabric-cicd, Terraform, or your own code.

Best forEdge cases, CI/CD, and anything the higher layers don't cover yet.
LAYER 4

Custom MCP server

Your tools · your guardrails

Wrap REST + your provisioning logic into reusable, tenant-shaped tools — REST-level control, MCP-level safety & reuse.

Best forProductizing a repeatable, governed multi-tenant factory.
5 Build your own

A custom MCP server is your scaling moat

The Fabric Core MCP Server already proves the pattern: an MCP server that "calls the Fabric REST API on your behalf." An ISV can build the same kind of server — tuned to their product, their tenant model, and their guardrails.

  • ▤
    Domain-shaped toolsExpose onboard_tenant, apply_branding, rotate_keys — not raw endpoints.
  • ⛨
    Guardrails by constructionBake in least-privilege auth, naming rules, and destructive-action blocks the agent can't bypass.
  • ♻
    Write once, reuse everywhereThe same tools serve every agent, CLI, and pipeline across your org.
  • ⚖
    ComposableRun it alongside the Modeling & Core MCP servers — agents pick the right tool per job.

8 Executive summary

Six decisions that define a Fabric-scale ISV

01

Name the layers precisely

Modeling MCP Server (model), Core MCP Server (platform), Report Authoring skill (visuals), REST APIs (foundation) — each has a distinct job.

02

Skills teach, MCP does

Skills carry the expertise; MCP servers execute against your data. Use them together, not interchangeably.

03

Drop to REST for control

When higher layers don't fit, the Fabric REST APIs — via fabric-cicd, Terraform, or the CLI — give you the full surface.

04

Wrap it in a custom MCP

Package REST + your provisioning logic into tenant-shaped tools for REST-level control with MCP-level safety and reuse.

05

Profiles do the scaling

Service principal profiles isolate each tenant while sharing capacity — reaching tens of thousands of tenants.

06

The pipeline absorbs growth

One template × one pipeline = N tenants. Consistency and governance become structural, not heroic.
