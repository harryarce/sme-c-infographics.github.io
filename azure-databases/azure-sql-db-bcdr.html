Four ways things can go wrong

Business continuity means planning for disruption. Azure SQL Database has a specific feature to address each of these scenarios.

Local hardware or software failure

A server, drive, or process fails in the datacenter hosting your database node.

The Vocabulary

RTO & RPO: the two numbers that matter

Every business continuity plan boils down to two questions: How long can we be down? and How much data can we afford to lose?

RTO

Recovery Time Objective

How long it takes for your application to fully recover after an unplanned event. In short: downtime.

Event Recovered

Clock starts at outage → ends when service resumes

RPO

Recovery Point Objective

How much data (measured in time) you can afford to lose. In short: data loss window.

High Availability with Zone Redundancy

Availability Zones are physically separate datacenters inside a single Azure region, each with independent power, cooling, and networking. Zone redundancy uses synchronous replication to keep replicas in at least two other zones continuously synchronized. When a zone fails, a replica that already has every committed transaction takes over automatically.

RTO
< 30s
RPO
0
Scope
Zone
  • Synchronous replication: every committed write exists on a replica before it completes, designed to deliver zero committed data loss on HA failover
  • This RPO of zero applies only to in-region HA failover. Backup-based recovery (PITR, geo-restore) always carries a data loss window equal to the time since the last log backup
  • Transparent failover within the region, with no app changes required
  • Higher SLA than the default enterprise-class SLA
  • Available across most Azure regions that support Availability Zones

Stronger Together

Failover Groups: Active Geo-Replication, Amplified

Active Geo-Replication handles the data. Failover Groups handle everything else: stable endpoints, multi-database orchestration, and policy-driven failover decisions. Same sub-60-second RTO, far less operational friction.

🔗

Connection strings that survive failover

Geo-Replication alone changes your connection string after failover because the new primary is on a different logical server. Failover Groups layer DNS CNAME records on top, so myfog.database.windows.net routes to the current primary. No redeploys, no config changes, no downtime from a stale endpoint.

📦

Your entire application, not just one table

Geo-Replication is configured per database. Failover Groups let you fail over an entire server's worth of databases as a single coordinated unit, keeping your application from running against a split state mid-failover.

🧭

A built-in read scale endpoint

A second endpoint, myfog.secondary.database.windows.net, routes read-only traffic to the current secondary. Route your reporting or analytics workloads there and keep the primary focused on writes.

🎛️

You decide when to fail over

Customer-managed failover policy (the recommended default) means a regional incident never triggers an automatic failover without your say. When you are ready, one command moves everything. Microsoft-managed policy is available for teams that want to delegate the decision.

📐

Zero-touch secondary provisioning

Add a database to the group and the secondary replica is created automatically in the right region, at the right service tier, and at the right compute size. One abstraction to manage instead of individual per-database replication links.

⚡

All the speed, none of the complexity

Underneath, it is still Geo-Replication doing the heavy lifting. You get the same sub-60-second RTO and near-zero RPO, but with the app-facing stability and operational structure that a production DR plan actually demands.
